Jiang types are read left to right, from the inner value outward. Each suffix wraps the type before it.

## Primitive Types

```c
Bool flag = true;
Int count = -123;
UInt size = 123;
UInt8 byte = 255;
Int16 small = -45;
Float value = 12.3;
Double precise = 132.54;
Char ch = 'a';
```

`Int` and `UInt` are pointer-sized integers. Use fixed-width types such as `Int32` or `UInt64` when ABI, file, or protocol layout needs an exact width.

Enum discriminants are stored as `Int32` by default unless the enum declares another
integer representation.

`()` is the Unit type. It is a zero-size value and is also used for functions with no meaningful returned data.

String literals are UTF-8 byte sequences. Their default borrowed view type is `UInt8[:0]&`, so the
compiler can expose null-terminated data for C-style boundaries while keeping the slice length
separate from the sentinel byte:

```c
UInt8[_] bytes = "jiang";
UInt8[:0]& view = "jiang";
```

String literals can still be assigned directly where the expected type is a plain `UInt8[]&`.
For non-literal sentinel slices, convert explicitly with `slice()` or a cast to `UInt8[]&`:

```c
UInt8[:0]& c_view = "jiang";
UInt8[]& plain = c_view.slice();
UInt8[]& forced = c_view$.as(UInt8[]&);
```

## Type Suffixes

Suffixes farther to the right wrap a wider layer:

```c
Int[2][3] matrix;
Int?[2][3] nullable_items;
Int?[2]?[3] nullable_rows;
```

Common suffix forms:

| Syntax | Meaning |
| --- | --- |
| `T!` | the current layer is mutable |
| `T?` | short form of `Option<T>` |
| `T^` | short form of `Box<T>` |
| `T&` | short form of `Reference<T>` |
| `T*` | short form of `RawPointer<T>` |
| `T[]` | unsized array pointee |
| `T[]&` | borrowed slice reference |
| `T[:S]` | sentinel-terminated unsized array pointee |
| `T[*]` | short form of `ManyPointer<T>` |
| `T[*:S]` | sentinel-terminated many raw pointer |
| `T[N]` | fixed-size array |
| `T[N:S]` | sentinel-terminated fixed-size array |
| `T@E` | short form of `Result<T, E>` in return position |

Within one layer, `?` must come before `!`: write `Int?!`, not `Int!?`, `Int??`, or `Int!!`.

## Type Inference

Use `_` as a placeholder for the right-hand side type. In a local binding, the compiler first
computes the initializer's natural type, ignores any outer mutability carried by the source binding,
and then applies the binding suffixes written after `_`:

```c
_ answer = 42;
_ name = "Jiang";
_ values = [1, 2, 3];
```

The suffixes after `_` belong to the new binding, not to the right-hand side:

| Binding pattern | Meaning |
| --- | --- |
| `_ name` | bind a new immutable value with the RHS type |
| `_! name` | bind a new mutable value with the RHS type |
| `_& name` | borrow the RHS place and bind an immutable `T&` |
| `_&! name` | borrow the RHS place and bind a mutable reference slot containing `T&` |

Use `_!` when the new binding itself must be reassignable:

```c
_! count = 0;
count = count + 1;
```

Inference preserves the expression's natural ownership/reference shape:

```c
Int^ make_value();

_ owner = make_value();  // owner: Int^
Int copied = make_value()$.get();
```

The right-hand side binding's outer mutability is intentionally ignored. `_` describes the type of
the value being bound; `!` after `_` describes whether the new binding is mutable:

```c
Int! source = 1;
_ copied = source;     // immutable binding, value type Int
_! mutable = source;   // mutable binding, value type Int
```

Use `_&` to infer a borrowed reference binding from a place expression:

```c
Int value = 1;
_& borrowed = value;  // borrowed: Int&
Int copied = borrowed$.get();
```

`_&` is a binding mode. It actively borrows the right-hand side place and infers the referenced
type. The source must be a place that can be borrowed; if the expression is already a reference, use
`_` instead of `_&`:

```c
Int& existing = value$.ref();
_ same_ref = existing;   // same_ref: Int&
// _& nested = existing; // error: `_&` does not borrow an existing reference again.
```

Use `_&!` when the reference binding itself must be reassignable. The `!` applies to the binding's
outer layer; it does not require the borrowed source to be mutable:

```c
Int first = 1;
Int second = 2;
_&! current = first;
current = second$.ref();
```

The same `_` binding patterns are used inside destructuring and payload patterns:

```c
(_& left, _! right) = pair;

if maybe is some _& item {
    use(item);
}
```

## Layered Mutability

`!` marks the mutability of the current layer. In a binding declaration, the outermost `!` marks the
binding slot itself, not the value type stored in that slot. Inside a compound type, `!` is part of
that nested value type:

```c
Bool! flag = true;     // mutable binding slot containing Bool
flag = false;

Int[3]! values = [1, 2, 3]; // mutable binding slot containing Int[3]
values = [4, 5, 6];

Int![3]! mutable_items = [1, 2, 3]; // mutable binding slot containing Int![3]
mutable_items[0] = 10;
mutable_items = [4, 5, 6];
```

For example, `Int! value` declares a mutable binding whose stored value type is `Int`; `Int![]`
uses `!` one layer deeper, so the array element type is mutable. `Int[]! value` is a mutable binding
whose stored value type is `Int[]`.

For aggregate values, a member is writable only when that member's own type is mutable.

```c
struct User {
    Int id;
    Int! age;
}

User user = User(id: 1, age: 18)
user.age = 19;
// user.id = 2; // error
```

## Arrays and Slices

Array length is part of the type:

```c
Int[3] values = [1, 2, 3];
Int[_] inferred = [1, 2, 3];
```

Nested arrays use repeated suffixes:

```c
Int[2][3] matrix = [[1, 2], [3, 4], [5, 6]];
```

`T[]` is an unsized array pointee. Use `T[]&` for a borrowed slice reference with runtime length:

```c
Int[_] values = [1, 2, 3];
Int[]& view = values[..];
```

Sentinel slice views such as `T[:S]&` carry an additional type-level sentinel guarantee. A
non-literal sentinel slice does not implicitly lose that guarantee when assigned to `T[]&`; write
`value.slice()` or `value$.as(T[]&)` to request a plain slice view.

## Pointers and References

```c
Int^ owner = new Int(42);
Int value = owner$.get();
owner$.free();
```

`T^`, `T&`, and `T*` do not auto-dereference in ordinary value contexts. Use `$.get()` for explicit reads. Member access is the exception: `owner.member` can pass through `T^`:

```c
Int value = 41;
Int& ref = value$.ref();
Int copied = ref$.get();

struct Box {
    Int value;
}

Box^ box = new Box(value: 42);
Int n = box.value;
```

## Optional

`T?` means the value may be `null`. It is equivalent to `Option<T>`:

```c
Int? maybe = 42;
Option<Int> explicit = 42;
Int? none = null;
```

Use optional chaining, `??`, or `is some`:

```c
_ field = user?.name;
Int value = maybe ?? 0;

if maybe is some payload {
    return payload;
} else {
    return 0;
}
```

## Errorable Returns

`T@E` is used in function and method return position. It is equivalent to `Result<T, E>` there:

```c
enum Err {
    bad = 1,
}

Int@Err ok() {
    return 42;
}

Int main() {
    return try ok() catch () => 0;
}
```

Use `throw expr;` to return the error side and `try expr catch (...) => fallback` to handle it.
