Jiang 使用 `T^` 表达拥有所有权的值，使用 `T&` / `T[]&` 表达不拥有资源的借用视图。
这一章集中说明资源如何移动、何时析构，以及引用如何避免悬垂。

## Copy、move 和析构

普通 `struct` 和 `union` 默认可以隐式 copy。`T^` 是内建资源类型，不能隐式 copy；
显式声明 `Movable` 的 nominal type 也不能隐式 copy。`T&`、`T[*]`、`T*` 和 `T[]&`
都是 non-owning view，字段中包含这些类型不影响 implicit copy。`!` 只是当前类型层级的可变性标记，
不改变所有权分类。

`T?` 的所有权分类跟随内部类型。`Int?` 可以像普通值一样隐式 copy；`Int^?` 包含拥有
所有权的 payload，因此是 `Movable`，不能隐式 copy，转移时也必须写 `$.move()`。

拥有所有权的值只能有一个当前所有者。把 `T^` 赋给另一个变量时，必须显式写 `$.move()`：

```c
Int^ a = new Int(42);
Int^ b = a$.move();

Int value = b$.get();
```

不能隐式复制或隐式转移所有权：

```c
Int^ a = new Int(42);
Int^ b = a; // 错误：T^ 不能隐式 copy，也不能通过普通赋值隐式 move。
```

显式 move 使用 `value$.move()`：

```c
Buffer^ a = new Buffer();
Buffer^ b = a$.move();

// a 已经失效，后续不能读取、写入、调用方法或再次 move。
```

move 之后继续使用源变量会报错：

```c
Int^ a = new Int(1);
Int^ b = a$.move();
Int value = a$.get(); // 错误：a 已经被 move。
```

普通值仍然可以 copy：

```c
Int a = 1;
Int b = a;
Int c = a + b;
```

直接或间接包含 `Movable` 字段的 nominal type 必须显式声明 `Movable`，并且不能隐式 copy：

```c
struct Box: Movable {
    Int^ value;
}

Box a = Box(value: new Int(1))
Box b = a; // 错误：Box 是 Movable，不能隐式 copy。
Box c = a$.move();
```

non-owning view 字段不会让类型变成 `Movable`：

```c
struct SliceView {
    UInt8[]& data;
}

SliceView a = SliceView(data: "abc")
SliceView b = a; // 允许：T[]& 不拥有资源，不影响 implicit copy。
```

析构规则：

- 只有 `Movable` 类型会自动 drop。
- `T^` 离开作用域时会析构指向的 `T`，并释放堆存储。
- `T?` 的自动析构由内部类型决定：`some` 会析构 payload，`none` 不做 payload 析构。
- `struct` 的 `T^` 字段会自动析构。
- `T&`、`T[*]`、`T*` 和 `T[]&` 字段不会自动释放。
- 如果 `struct` 有自定义 `deinit`，先执行自定义 `deinit`，再自动析构拥有所有权的字段。

## 引用规则

引用不拥有目标对象。创建引用时，引用的使用范围不能超过被引用的值：

```c
Int value = 10;
Int& ref = value$.ref();
Int copied = ref$.get();
```

在局部绑定和 pattern 中，`_&` 是一种推导借用绑定的简写：

```c
Int value = 10;
_& ref = value; // 生命周期规则等同于 value$.ref()
```

同一个 binding mode 也可以用于 optional 和 union 的 payload pattern。它借用 payload，
不会隐式 copy 或 move payload：

```c
Int read_ref(Int& value) {
    value$.get()
}

Int read_optional(Int? value) {
    if value is some _& item {
        return read_ref(item);
    }
    return 0;
}
```

返回局部变量引用是不允许的，因为函数返回后局部变量已经不存在：

```c
Int& bad_ref() {
    Int value = 10;
    return value$.ref(); // 错误：返回值引用了局部变量。
}
```

如果要返回一个值，直接返回值本身：

```c
Int make_value() {
    Int value = 10;
    return value;
}
```

如果要返回一个堆上对象的所有权，返回 `T^`：

```c
Int^ make_owner() {
    return new Int(10);
}
```

Jiang 的引用规则不把“共享引用”和“可变引用”设计成两套互斥能力。它关注的是引用是否悬垂；
数据竞争和更严格的别名规则不属于当前 borrow check 的核心目标。

## 生命周期约束

`@life(a > b)` 表示 `a` 的目标生命周期必须长于 `b`。这套规则用于避免悬垂引用；
它不检查共享/可变别名，也不保证数据竞争安全。

存储 `T&` 或 `T[]&` 字段时，目标对象的生命周期必须覆盖包含该字段的值。跨函数或存储到字段的关系可以用 `@life(...)` 表达：

```c
@life(data > self)
struct Slice {
    UInt8[]& data;
    Int len;
}

@life(buffer > return)
Slice make_slice(Buffer& buffer);
```

函数返回引用时有一条默认推断规则：如果返回引用只可能来自同一种输入来源，那么该来源的
lifetime 默认覆盖 `return`。这里的输入来源可以是某个参数，也可以是 `self`。

```c
Int& first(Int& value) {
    return value; // 允许：返回来源只可能是 value。
}

struct Pair {
    Int left;
    Int right;

    Int& choose(Bool use_left) {
        if use_left {
            return self.left$.ref();
        }
        return self.right$.ref(); // 允许：两个分支都来自同一个来源 self。
    }
}
```

如果返回引用可能来自多个输入来源，就必须显式写出所有允许来源。显式 `@life(... > return)`
出现后，borrow check 只允许标注中的来源：

```c
@life(left > return, right > return)
Int& pick(Int& left, Int& right, Bool use_left) {
    if use_left {
        return left;
    }
    return right;
}
```

缺少标注时会失败：

```c
Int& bad_pick(Int& left, Int& right, Bool use_left) {
    if use_left {
        return left;
    }
    return right; // 错误：返回来源可能是 left 或 right，必须显式 @life。
}
```

正例：结构体字段保存外部 slice 时，用 `@life(data > self)` 表示 `data` 指向的存储必须长于结构体本身：

```c
@life(data > self)
struct ByteView {
    UInt8[]& data;
}

ByteView view(UInt8[]& data) {
    return ByteView(data: data)
}
```

反例：把局部数组切片存进返回值会产生悬垂引用：

```c
@life(data > self)
struct ByteView {
    UInt8[]& data;
}

ByteView bad_view() {
    UInt8[_] local = "abc";
    return ByteView(data: local[..]) // 错误：local 的生命周期短于返回值。
}
```

公共 API、跨函数调用、返回含引用字段的聚合值，仍建议显式写出返回来源关系：

```c
@life(data > return)
UInt8[]& first_two(UInt8[]& data) {
    return data[0..2];
}
```

如果没有足够长的输入生命周期，应该返回拥有所有权的值，而不是返回引用视图。
